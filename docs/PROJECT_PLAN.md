# 小说扫评 Agent 项目计划

更新日期：2026-07-18

## 1. 项目目标

构建一个面向网络小说读者的本地优先扫评 Agent：用户导入自己的小说文件，选择关心的“雷点”和“郁闷点”，应用完成全书扫描，并把每条结论定位回原书章节及可复核原文。

产品边界如下：

- 首发平台为 Windows 和 Android；不规划 iOS，也不把应用商店上架作为首发前置条件。
- 同一套 Rust 扫描核心、导入契约和规则数据服务两个客户端，避免两端结论漂移。
- 用户自带模型凭据（BYOK）。ChatGPT Plus/Codex 订阅用于开发协作，不等同于成品调用模型 API 的额度。
- 未配置模型凭据时，仍可导入、管理书籍、编辑规则和查看历史结果；确定性测试提供器只用于测试和演示，不冒充 AI 模型。
- 社区规则包是特定社区语境下的可选检查清单，不代表普遍文学评价，也不对作者或读者作价值判断。

## 2. 可验收的产品结果

第一版发布应同时满足：

1. Windows 安装包与 Android 签名 APK 来自同一 GitHub 仓库和可重复的发布流程。
2. 用户能导入不止 TXT 的多种文件；界面必须如实区分“可解析”“仅识别”“需要 OCR/转换”和“不支持”。
3. 每本书可独立选择规则、开关规则，并覆盖雷点/郁闷点等级。
4. 长篇扫描可暂停、继续和崩溃恢复；修改过的源文件不会误用旧检查点。
5. 已确认命中必须带章节、格式相关位置和从原文重建的精确摘录；无法回证的模型判断只能标为“疑似”。
6. 用户可切换模型提供商，密钥不进入前端状态、日志、SQLite 或 Git 仓库。
7. 规则包保留版本、来源、核验状态和变更记录；未核验条目默认关闭。

## 3. 分阶段计划表

阶段状态以 GitHub issue/milestone 为准；下表中的“完成门槛”是进入下一阶段前必须通过的检查，而不是日期承诺。

| 阶段 | 交付重点 | 完成门槛 |
| --- | --- | --- |
| S1 跨端基础骨架 | React/TypeScript 响应式界面、Tauri 2 桌面壳、共享 Rust workspace、导入与模型提供器契约、种子规则包、基础 CI | 前端测试与构建通过；规则包校验通过；Windows CI 完成 Rust fmt/check/test；架构和安全边界有文档 |
| S2 多格式导入 | TXT、Markdown 的完整导入；EPUB、DOCX、HTML 文本导入；文本型 PDF；统一章节与来源锚点 | 每种“可解析”格式都有正常、损坏和超限样本；章节和摘录可回到源位置；界面不把待实现格式显示为已支持 |
| S3 扫描与长上下文 | 分章流水线、上下文快照、实体/关系/事件账本、未决候选、断点续扫、用量预算 | 超长样书能在固定上下文预算内扫完全书；暂停/恢复结果稳定；源文件变化时检查点失效 |
| S4 多模型 BYOK | 提供器注册表、OpenAI-compatible 适配器及所选原生适配器、超时/重试/限流、结构化输出校验 | 可切换至少两个真实提供器；密钥仅经原生安全存储读取；网络失败不会丢失进度；无密钥也可进入应用 |
| S5 社区规则核验 | 对 yy小说吧种子条目逐项核对原帖和社区用法；用户预设；规则包升级/迁移 | 每个默认启用条目至少有可访问来源和复核记录；同名跨分类概念保持独立；未核验条目默认关闭 |
| S6 Windows 与 Android 产品化 | Windows 安装/升级；审查并提交必要的 Android scaffold/原生定制；实现 SAF 文件导入、前台长任务、Keystore 密钥封装和响应式交互 | Windows 真实安装冒烟通过；Android 真机完成导入—暂停—恢复—查看证据闭环；无 iOS 构建依赖 |
| S7 质量与发布 | 隐私说明、威胁检查、性能与可访问性、崩溃恢复、GitHub Release | CI 全绿；测试样书无内容泄露；发布资产带校验值和变更说明；已知限制在 README/Release 中公开 |

## 4. 多格式能力分级

“识别格式”不等于“能抽取正文”。应用、文档和测试统一使用以下状态：

| 等级 | 含义 | 格式与目标位置 | 当前基础状态 |
| --- | --- | --- | --- |
| A：可解析且可回证 | 能生成章节、正文和稳定来源锚点，可投入扫描 | TXT（章节/行号/UTF-8 字节范围）、Markdown（标题/行号） | TXT、Markdown 基础导入已进入核心；GBK/GB18030 仍需补齐并测试 |
| B：计划内结构化解析 | 格式可识别，但只有解析器和回证测试完成后才能标为可用 | EPUB（spine、资源路径、CFI/段落）、DOCX（标题层级、段落号）、HTML（资源与 DOM 锚点）、文本型 PDF（页码与文本块） | 已有格式识别/契约，解析器按 S2 交付 |
| C：转换或受限支持 | 依赖额外解析、无 DRM 或资源限制；必须提示降级 | MOBI/AZW3（仅无 DRM）、ZIP/7Z 内的受支持文档、复杂排版 PDF | 计划内；压缩包需防路径穿越和解压炸弹 |
| D：需要 OCR/外部转换 | 原文件没有可靠文本层，结果的位置精度和成本不同 | 扫描版 PDF、图片；旧版二进制 DOC | OCR 作为独立能力展示；旧 DOC 首版提示另存为 DOCX/TXT/PDF |

每次导入先做内容签名、媒体类型和扩展名的交叉识别，再返回明确能力状态。不能因为扩展名在列表中就静默按文本读取。

## 5. 核心工作流

1. 用户通过 Windows 文件选择器或 Android Storage Access Framework 选择文档。
2. 导入层识别格式、抽取章节、保留格式相关来源锚点，并计算文档与章节指纹。
3. 用户选择规则包、规则开关及严重等级，然后选择模型提供器和预算。
4. 扫描器逐章提交“本章原文 + 有界上下文快照 + 选中规则”，验证模型返回的结构和范围。
5. 核心只能从已导入原文重建证据摘录。范围合法且规则有效时可确认；否则保留为疑似或拒绝。
6. 每章事务性保存命中、用量和上下文检查点，支持暂停与继续。
7. 结果页按规则、等级、人物和章节筛选，并允许跳回原文位置。

## 6. 测试与发布原则

- 单元测试使用小型、原创或明确授权的夹具，不把用户小说提交进仓库。
- 前端测试覆盖规则选择、能力状态、证据状态和移动布局的关键逻辑。
- Rust 测试覆盖格式识别、编码、章节切分、上下文预算、提供器返回校验和断点恢复。
- 规则包作为数据产品独立校验：数量不是唯一门槛，还要检查唯一 ID、来源引用、核验状态、默认开关和跨分类概念。
- Android 构建是显式手动触发的验证流程，runner 固定安装 API 36、Build Tools 36.0.0 与 NDK r29 `29.0.14206865`。提交中没有原生 scaffold 时，CI 使用 Tauri 的 `android init --ci --skip-targets-install` 临时生成后再尝试 debug 构建。
- CI 临时 scaffold 会随 runner 销毁，只验证基础生成/编译路径；它不代表 application ID、SDK 策略、权限、SAF、Keystore、长任务或真机体验已经产品化。必要的原生定制在 S6 经人工复核后提交。
- GitHub Release 首发可分发 Windows 安装包和侧载 APK；是否进入第三方商店以后再决定。
