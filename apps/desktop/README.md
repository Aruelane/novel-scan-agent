# Tauri shell

This directory owns the native shell. The React UI remains in `../client`, while
the platform-independent scanner and import contracts remain in the workspace
Rust crates.

## Supported targets

- Windows desktop
- Android (APK distribution is sufficient for the first release)
- iOS is intentionally out of scope. The capability file is restricted to
  `windows` and `android`, and no iOS script or generated Xcode project is kept.

`src-tauri/gen/android` is generated by Tauri and ignored by Git. After the
Android SDK/NDK prerequisites are installed, run `npm run android:init` once
before `npm run android:dev` or `npm run android:build`.

## Commands

Install the shell CLI from this directory and the frontend dependencies from
`../client`:

```powershell
npm install
npm --prefix ../client install
```

Generate the binary Windows/Android icon set from the versioned, text-reviewable
SVG source before the first native build:

```powershell
npm run icons
```

This runs Tauri CLI `icon` against `assets/app-icon.svg`, writes the Windows
and Android resources to `src-tauri/icons`, and removes the unrelated Apple
outputs generated by the cross-platform CLI.

Then use:

```powershell
npm run dev
npm run build
npm run android:init
npm run android:dev
npm run android:build
```

The scripts invoke Node entry points directly and never interpolate the working
directory into a shell command. This matters because the local repository path
may contain shell metacharacters such as `&&`.

## Native boundary

`import_capabilities` serializes the registry supplied by `novel-import` and
also maps each format to the shared `novel-core::DocumentFormat`. The same Rust
library exposes `run()` as the desktop entry point and as a Tauri mobile entry
point, so Android does not require a Python sidecar or child process.

The WebView capability is restricted to `allow-import-capabilities`. SQLite
access, file dialogs, and secret storage are handled exclusively through the
Rust command layer; the frontend has no direct SQL or filesystem permissions.

## Local database and credentials

The SQL plugin preloads `sqlite:novel-scout.db` and atomically applies
`migrations/0001_initial.sql`. The first schema stores books, chapters, scan
jobs, selected rules, resumable checkpoints, findings, exact evidence, and
non-secret provider profile metadata.

API keys are not SQLite columns. `provider_profiles.credential_ref` only accepts
an opaque `secret-ref:` handle. A later secret-store adapter must resolve that
handle through Windows Credential Manager or Android Keystore; plaintext keys,
tokens, passwords, and authorization headers must never be written to this
database.
